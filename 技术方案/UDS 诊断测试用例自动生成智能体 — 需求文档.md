## 1. 引言

### 1.1 编写目的

本文档定义"UDS 诊断测试用例自动生成智能体"的完整需求，作为开发、测试和验收的基准依据。

### 1.2 术语与缩略语

| 术语 | 缩写 | 定义 |
| --- | --- | --- |
| Unified Diagnostic Services | UDS | 统一诊断服务，ISO 14229 标准定义的汽车诊断协议 |
| Service Identifier | SID | 服务标识符，如 0x10、0x22 |
| Negative Response Code | NRC | 否定响应码，表示诊断请求失败的原因 |
| Diagnostic Trouble Code | DTC | 诊断故障码，标识 ECU 检测到的具体故障 |
| Data Identifier | DID | 数据标识符，用于读写 ECU 内部数据 |
| Routine Identifier | RID | 例程标识符，用于控制 ECU 执行特定例程 |
| Suppression of Positive Response Message Indication Bit | SPRMIB | 正响应抑制位，置位后 ECU 不返回正响应 |
| Computer Aided Programming Language | CAPL | Vector CANoe/CANalyzer 的脚本语言 |
| ECU | Electronic Control Unit | 电子控制单元 |

## 2. 业务需求

### 2.1 业务背景

汽车电子 ECU 的 UDS 诊断测试用例编写是一项高重复、高复杂度、强依赖经验的工作。当前面临：

1. **周期长**：11 个 UDS 服务的人工编写需数周，严重影响项目交付周期
2. **质量不稳定**：不同工程师对协议规则理解深度不同，导致用例覆盖粒度差异大
3. **难以复用**：不同 ECU 项目参数不同，每次需从零编写，无法批量迁移
4. **人才瓶颈**：资深 UDS 测试工程师稀缺，新人培训周期长

### 2.2 业务目标

| 编号 | 目标 | 可度量指标 |
| --- | --- | --- |
| BG-01 | 测试用例生成自动化 | 11 个 UDS 服务全覆盖，单服务生成时间 ≤ 1 分钟 |
| BG-02 | 规则准确率达标 | 与人工编写的黄金标准比对，准确率 ≥ 95% |
| BG-03 | 消除人为差异 | 同一输入 Excel，不同时间生成的用例一致性 ≥ 99% |
| BG-04 | 适应不同项目 | 支持格式不固定的 Excel 输入，不要求客户调整文件格式 |
| BG-05 | 低成本集成 | 集成到公司现有低代码平台，不额外开发前后端 |

### 2.3 用户角色

| 角色 | 描述 | 使用场景 |
| --- | --- | --- |
| UDS 测试工程师 | 日常编写 UDS 测试用例的一线人员 | 上传 ECU 参数 Excel → 一键生成全量用例 → 审核/微调 → 导出 |
| 测试主管 | 管理测试团队，关注用例质量和交付进度 | 批量生成所有服务的用例 → 抽检准确性 → 验收 |
| ECU 项目经理 | 负责 ECU 项目整体进度 | 快速获取全部诊断测试用例，缩短测试准备周期 |

## 3. 功能需求

### 3.1 核心功能概览

### 3.2 F1: Excel 文件解析

**描述**：接收用户上传的 ECU 诊断参数 Excel 文件，转换为结构化文本供后续处理。

| 需求编号 | 需求描述 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| F1-01 | 支持 .xlsx 格式的 Excel 文件 | P0 | 标准 xlsx 文件可正常读取 |
| F1-02 | 支持 .xls 格式的 Excel 文件（旧格式） | P1 | 能转换并读取旧格式文件 |
| F1-03 | 自动识别并读取所有 Sheet 页 | P0 | 无论 Sheet 命名规则如何，都能读取内容 |
| F1-04 | 保留合并单元格的值（左上角） | P0 | 合并单元格信息不丢失 |
| F1-05 | 读取公式计算结果而非公式本身 | P0 | data_only=True 模式 |
| F1-06 | 单 Sheet 最大支持 200 行 | P0 | 超过 200 行的 Sheet 截断并给出警告 |
| F1-07 | 检测并拒绝含图片/宏的异常文件 | P1 | 给出明确错误提示 |
| F1-08 | 文件大小限制：≤ 20MB | P1 | 超出限制时给出提示 |

### 3.3 F2: 参数智能提取

**描述**：使用 LLM 从 Excel 文本中语义理解并提取结构化参数，不依赖固定的列号或 Sheet 名。

| 需求编号 | 需求描述 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| F2-01 | 提取 ECU 基本参数：ECU名称、协议、CAN ID（请求/响应/功能寻址）、P2/P2*/S3 超时时间 | P0 | 所有基本参数字段提取完整 |
| F2-02 | 提取服务支持矩阵：每个子功能的 Support/SPRMIB/Session/Access Level/NRC | P0 | 子功能列表完整，每个字段均有值 |
| F2-03 | 按 Service ID 筛选，仅提取目标服务相关的参数 | P0 | 不提取无关服务的参数 |
| F2-04 | 语义字段匹配：按字段含义（如"Service ID"、"Subfunction"）而非固定列号定位 | P0 | Sheet 名/列顺序变化后仍能正确提取 |
| F2-05 | 布尔值标准化：x/X/Y/Yes → true；N/No/空白/横杠 → false | P0 | 输出 JSON 中布尔字段格式统一 |
| F2-06 | 数值标准化：P2/P2*/S3 转毫秒整数，CAN ID 转十六进制，NRC 用逗号分隔 | P0 | 数值格式统一 |
| F2-07 | 提取 DID 列表（仅 0x22/0x2E 需要）：DID 编号/字节长度/读写支持/访问等级/会话 | P0 | DID 列表与 Excel 中定义一致 |
| F2-08 | 提取 DTC 列表（仅 0x14/0x19 需要）：DTC 编号/名称/状态位/触发条件 | P0 | DTC 列表与 Excel 中定义一致 |
| F2-09 | 提取 RID 列表（仅 0x31 需要）：RID 编号/子功能/访问等级/请求响应参数 | P0 | RID 列表与 Excel 中定义一致 |
| F2-10 | 提取 K 列业务规则（最高优先级），如存在输出模板文件 | P1 | 客户定制规则被正确识别和提取 |
| F2-11 | 支持复位时间参数（reset_time_support/byte_length），用于 0x11 服务 | P1 | 正确识别 ECU 是否支持复位时间 |
| F2-12 | 找不到的字段填充空字符串，不报错 | P0 | 缺失字段不中断流程 |
| F2-13 | JSON 输出格式校验：自动重试一次解析失败的输出 | P0 | 连续 2 次失败才返回错误 |
| F2-14 | 支持大型 DID 表（100+条）的分批提取 | P1 | 分批后汇总合并，不遗漏 |

| --- | --- | --- | --- |
| F3-01 | 覆盖全部 11 个 UDS 服务：0x10/0x11/0x14/0x19/0x22/0x27/0x28/0x2E/0x31/0x3E/0x85 | P0 | 每个服务均可独立生成用例 |
| F3-02 | 按"分类"结构组织用例：Session Layer / SPRMIB / Secure Access / 服务特有分类 / Incorrect Command | P0 | 每个服务的所有分类均有用例 |
| F3-03 | Case ID 格式：`Diag_0x{SID}_{Phy|Fun}_{序号}` | P0 | 生成的 Case ID 格式 100% 正确 |
| F3-04 | 物理寻址用例编号 < 功能寻址用例编号 | P1 | Phy 用例先于 Fun 用例 |
| F3-05 | 正响应 SID 正确：原 SID + 0x40 | P0 | 正响应字节无错误 |
| F3-06 | 负响应格式正确：`7F {原SID} {NRC}` | P0 | 负响应 SID 匹配 |
| F3-07 | NRC 使用仅限参数表中列出的值 | P0 | 不生成未在 NRC 列表中的否定响应码 |
| F3-08 | 每个 Send 步骤有对应的 Check 步骤（Delay/AndCheckResp 豁免） | P0 | 步骤-Check 配对率 100% |
| F3-09 | 使用标准 CAPL 函数 API（Send DiagBy/Check DiagData/Send Security 等） | P0 | 仅使用 `汽车电子函数接口定义.docx` 中定义的 21 个函数 |
| F3-10 | 会话切换路径正确：Default → Extended/Programming 的标准进入路径 | P0 | 会话路径符合协议规范 |
| F3-11 | 安全解锁流程正确：Seed/Key 配对、等级映射（奇数=Seed，偶数=Key） | P0 | Seed/Key 子功能编号正确 |
| F3-12 | SPRMIB 抑制位正确：0x80 + Subfunction，抑制后无正响应 | P0 | SPRMIB 测试步骤和 Check 正确 |
| F3-13 | K 列业务规则优先级最高，覆盖默认规则 | P0 | 客户定制规则被优先应用 |
| F3-14 | 支持 App 和 BootLoader 两个软件域的用例生成 | P1 | 按 software_domain 参数切换会话/行为规则 |
| F3-15 | 输出用例文本包含：Case ID、Case 描述、测试步骤、预期结果 | P0 | 输出格式完整 |

### 3.5 F4: 输出校验

**描述**：对 LLM 生成的测试用例进行自动校验，识别格式错误和逻辑缺陷。

| 需求编号 | 需求描述 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| F4-01 | Case ID 格式校验：匹配 `Diag_0x[0-9A-F]{2}_(Phy\|Fun)_\d{3}` | P0 | 格式错误的 Case ID 被识别为 error |
| F4-02 | Case ID 唯一性校验：无重复 ID | P0 | 重复 ID 被识别为 error |
| F4-03 | NRC SID 匹配校验：`7F` 后必须跟原 SID | P0 | SID 不匹配被识别为 error |
| F4-04 | 正响应 SID 校验：正响应 = 原 SID + 0x40 | P0 | 正响应字节错误被识别为 error |
| F4-05 | 步骤-Check 配对校验 | P1 | 未配对的步骤被识别为 warning |
| F4-06 | 分类完整性校验：检查各分类是否都有用例 | P1 | 缺失分类被识别为 warning |
| F4-07 | 物理寻址先于功能寻址校验 | P1 | 顺序错误被识别为 warning |
| F4-08 | 校验结果分级：error（阻断性）和 warning（提示性） | P0 | errors 和 warnings 分别输出 |
| F4-09 | valid 判定：仅当 errors 为空时 valid=true | P0 | 逻辑正确 |
| F4-10 | 输出用例总数统计 | P1 | case_count 字段准确 |

## 4. 各服务特有功能需求

### 4.1 各服务生成分类汇总

| 服务 | 服务名称 | 分类数 | 特有分类 |
| --- | --- | --- | --- |
| 0x10 | DiagnosticSessionControl | 7 | S3 Server Timer（核心分类） |
| 0x11 | ECUReset | 7 | Reset Effect（客户扩展：服务/安全/通信/故障复位） |
| 0x14 | ClearDiagnosticInformation | 5 | Clear DTC Function（5 场景：单/多 DTC 清除、无 DTC 清除、重新记录） |
| 0x19 | ReadDTCInformation | 4 | DTC Read Function（5 子功能：01/02/04/06/0A 各有独立响应格式） |
| 0x22 | ReadDataByIdentifier | 4 | DID Range Test（DID 遍历：排除支持的 DID，验证不支持的 DID 返回 NRC 0x31） |
| 0x27 | SecurityAccess | 8 | Security Access Mechanism（错误密钥/最大尝试/延时计时器/FAAflag）；Security Request Process（已解锁态重请求） |
| 0x28 | CommunicationControl | 7 | Communication Control Function（通信控制 + MonitoredMsgId 验证）；双遍历（子功能+通信类型） |
| 0x2E | WriteDataByIdentifier | 4 | DID Storage Test（写后回读验证 + 掉电持久性） |
| 0x31 | RoutineControl | 5 | Routine Condition Test（前置条件：电压/状态检查）；RID Range Test |
| 0x3E | TesterPresent | 7 | S3 Server Timer（4 场景：会话保持/超时/S3 后操作/S3 中 TesterPresent） |
| 0x85 | ControlDTCSetting | 7 | DTC Setting Function（8 场景：开启/关闭/跨服务协作 0x14+0x19） |

### 4.2 跨服务依赖

| 服务 | 依赖的服务 | 说明 |
| --- | --- | --- |
| 0x14 | 0x19 | 清除 DTC 后需用 0x19 验证清除效果 |
| 0x85 | 0x14 + 0x19 | 关闭 DTC 记录后需制造故障→清除→读取验证 |
| 0x2E | 0x22 | 写 DID 后需用 0x22 回读验证写入结果 |
| 0x3E | 0x22 (DID F186) | S3 超时验证需读取 DID F186 确认会话状态 |
| 0x27 | 0x11 | 安全锁定机制验证可能涉及 ECU 复位 |

## 6. 接口需求

### 6.1 输入规范

#### 6.1.1 必需输入

| 参数 | 类型 | 格式 | 说明 |
| --- | --- | --- | --- |
| input_excel_path | str | 文件路径 | ECU 诊断参数 Excel 文件（.xlsx/.xls） |
| service_id | str | "0x10" ~ "0x85" | 目标服务 ID |
| llm_base_url | str | URL | LLM 推理服务地址 |
| llm_model | str | 模型名 | 如 "Qwen3-32B" |

#### 6.1.2 可选输入

| 参数 | 类型 | 默认值 | 说明 |

| software_domain | str | "App" | 软件域："App" 或 "BootLoader" |

#### 6.1.3 Excel 文件语义字段要求

以下字段按**语义名称**识别（不依赖固定列号或 Sheet 名），至少需包含：

| 语义字段 | 必需 | 说明 |
| --- | --- | --- |
| Service ID | 是 | UDS 服务标识（0x10/0x11/.../0x85） |
| Subfunction | 是 | 服务子功能编号 |
| Support | 是 | 子功能是否支持（Y/N） |
| SPRMIB | 是 | 正响应抑制位支持 |
| Physical Request | 是 | 物理寻址支持 |
| Functional Request | 是 | 功能寻址支持 |
| Session (Default/Extended/Programming) | 是 | 各会话下的支持状态 |
| Access Level | 否 | 安全访问等级（部分服务不需要） |
| Negative Response Codes | 是 | 支持的否定响应码 |
| CAN ID (Req/Resp/Func) | 是 | 诊断 CAN ID |
| P2/P2*/S3 | 是 | 超时时间参数 |
| DID Number / Byte Length / R/W Support | 条件必需 | 0x22/0x2E 服务必需 |
| DTC Number / Status Mask | 条件必需 | 0x14/0x19 服务必需 |
| RID Number / Subfunction | 条件必需 | 0x31 服务必需 |

## 8. 验收标准

### 8.1 功能验收

| 验收项 | 验收方法 | 通过标准 |

| 11 个服务全部可生成 | 逐一调用 generate() 接口 | 每个服务均返回 valid=true 或仅有 warnings |
